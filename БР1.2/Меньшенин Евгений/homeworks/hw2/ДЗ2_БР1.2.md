# Домашняя работа №2: Проектирование и технический дизайн API

## Вариант проекта: Приложение для бронирования столиков в ресторанах

## 1. Функциональные требования к бэкенд-приложению

### Для клиентов:
1. **Регистрация и аутентификация пользователей**
   - Создание учетной записи с email и паролем
   - Вход в систему с получением токена доступа
   - Выход из системы

2. **Поиск и просмотр ресторанов**
   - Получение списка всех ресторанов с пагинацией
   - Фильтрация ресторанов по типу кухни, местоположению, рейтингу
   - Поиск ресторанов по названию
   - Просмотр детальной информации о ресторане

3. **Управление бронированиями**
   - Создание бронирования столика на определенное время
   - Просмотр списка своих бронирований
   - Просмотр детальной информации о конкретном бронировании
   - Изменение времени/даты бронирования
   - Отмена бронирования

4. **Отзывы и рейтинги**
   - Оставление отзывов о посещенных ресторанах
   - Просмотр отзывов других пользователей

### Для владельцев ресторанов:
1. **Управление рестораном**
   - Добавление и редактирование информации о ресторане
   - Управление меню ресторана

2. **Управление столиками**
   - Добавление, редактирование и удаление столиков
   - Установка вместимости и расположения столиков

3. **Управление бронированиями**
   - Просмотр всех бронирований своего ресторана
   - Подтверждение или отклонение бронирований
   - Отмена бронирований по инициативе ресторана

### Для администраторов:
1. **Модерация контента**
   - Просмотр и удаление неуместных отзывов
   - Блокировка пользователей при нарушении правил

2. **Управление пользователями**
   - Просмотр списка всех пользователей
   - Назначение ролей пользователям

## 2. Нефункциональные требования

### Производительность:
- Время отклика API не должно превышать 500 мс для 95% запросов
- Система должна поддерживать одновременную работу до 1000 активных пользователей
- Время отклика при пиковой нагрузке не должно превышать 1 секунду

### Масштабируемость:
- Архитектура должна поддерживать горизонтальное масштабирование
- Возможность добавления новых серверов без остановки сервиса
- Поддержка увеличения количества ресторанов и пользователей

### Безопасность:
- Все данные передаются по протоколу HTTPS
- Аутентификация пользователей через JWT токены
- Защита от SQL-инъекций и XSS атак
- Ограничение количества запросов от одного IP адреса
- Хэширование паролей пользователей

### Надежность:
- Время доступности системы не менее 99.9% (менее 8.7 часов простоя в год)
- Регулярное резервное копирование данных
- Мониторинг состояния системы и автоматическое уведомление администраторов о сбоях

### Документация:
- Полная документация API в формате OpenAPI
- Примеры использования всех эндпоинтов
- Четкое описание ошибок и кодов ответов

## 3. Формат реализации API

Для реализации API будет использован **REST API** 

## 4. Проектирование эндпоинтов API

Все эндпоинты API спроектированы с учетом RESTful принципов:

### Аутентификация и пользователи:
- `POST /auth/register` - Регистрация нового пользователя
- `POST /auth/login` - Аутентификация пользователя
- `GET /users/me` - Получение информации о текущем пользователе
- `PUT /users/me` - Обновление информации о текущем пользователе

### Рестораны:
- `GET /restaurants` - Получение списка ресторанов с фильтрацией
- `GET /restaurants/{id}` - Получение детальной информации о ресторане
- `GET /restaurants/{id}/menu` - Получение меню ресторана
- `GET /restaurants/{id}/reviews` - Получение отзывов о ресторане

### Столики:
- `GET /restaurants/{id}/tables` - Получение списка столиков в ресторане
- `GET /tables/{id}/availability` - Получение информации о доступности столика

### Бронирования:
- `POST /bookings` - Создание нового бронирования
- `GET /bookings` - Получение списка бронирований текущего пользователя
- `GET /bookings/{id}` - Получение детальной информации о бронировании
- `PUT /bookings/{id}` - Обновление бронирования
- `DELETE /bookings/{id}` - Отмена бронирования

### Отзывы:
- `POST /restaurants/{id}/reviews` - Добавление отзыва о ресторане
- `PUT /reviews/{id}` - Обновление своего отзыва
- `DELETE /reviews/{id}` - Удаление своего отзыва

## 5. Тело запросов и ответов

Для каждого эндпоинта спроектированы структуры запросов и ответов с учетом следующих принципов:

1. **Консистентность**: Все запросы и ответы следуют единому стилю именования (camelCase)
2. **Полная документация**: Каждое поле имеет описание и тип данных
3. **Валидация**: Все обязательные поля помечены как required
4. **Ограничения**: Установлены ограничения на длину строк, диапазоны чисел и т.д.

## 6. Возможные ошибки и коды ответов

Для обеспечения четкого взаимодействия между клиентом и сервером определены стандартные коды ошибок:

- `200 OK` - Успешный запрос
- `201 Created` - Ресурс успешно создан
- `204 No Content` - Запрос выполнен успешно, содержимое отсутствует
- `400 Bad Request` - Некорректный запрос (неправильные параметры)
- `401 Unauthorized` - Пользователь не аутентифицирован
- `403 Forbidden` - Доступ запрещен
- `404 Not Found` - Ресурс не найден
- `409 Conflict` - Конфликт данных (например, дублирование email)
- `422 Unprocessable Entity` - Данные прошли валидацию, но не могут быть обработаны
- `429 Too Many Requests` - Превышено количество запросов
- `500 Internal Server Error` - Внутренняя ошибка сервера
- `503 Service Unavailable` - Сервис временно недоступен

## 7. Заключение

API было спроектировано с учетом всех функциональных и нефункциональных требований к приложению. Структура эндпоинтов следует RESTful принципам, что обеспечивает интуитивно понятное взаимодействие. Полная спецификация в формате OpenAPI позволяет легко генерировать клиентские библиотеки и документацию. Спроектированная архитектура обеспечивает масштабируемость и надежность системы при высоких нагрузках.
